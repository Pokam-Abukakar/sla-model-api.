SLA Breach Prediction — Architecture & Workflows

GitHub: github.com/Pokam-Abukakar/sla-model-api  ·  API: sla-model-api-1.onrender.com/docs

1 · System Architecture

Every Jira ticket creation or update fires a webhook to n8n. The orchestration layer extracts the ticket fields, calls the XGBoost model API on Render, and evaluates the returned breach probability. Tickets scored as HIGH risk trigger two simultaneous actions: a label and risk score written back to Jira, and an HTML email alert sent to the service desk manager.

Figure 1: Four-layer system architecture. Dashed arrow = model response back to n8n.

2 · n8n SLA Breach Scoring Workflow

The 7-node n8n workflow runs automatically on every ticket event. The Code node maps Jira fields into the model API payload. The HTTP Request node calls POST /predict and receives a breach probability. The IF node branches on risk level — HIGH tickets proceed through nodes 5, 6, and 7 to trigger escalation; LOW and MEDIUM tickets stop silently with no agent disruption.

Figure 2: 7-node n8n workflow. TRUE branch escalates; FALSE terminates silently.

3 · Voice Ticket Creation Workflow

A caller dials a Twilio phone number which routes the call to an ElevenLabs AI voice agent. The agent collects the issue description, urgency, and department through natural conversation, then calls an n8n webhook via its create_ticket tool. n8n creates the Jira ticket and returns the ticket key, which the agent reads back to the caller. The new ticket immediately triggers the SLA breach scoring workflow above.

Figure 3: Voice ticket flow. Dashed arrows = ticket key returned to agent then read back to caller. New ticket triggers Figure 2 automatically.
